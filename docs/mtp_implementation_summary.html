Overview

This document summarizes the initial MTP speculative decoding implementation for Qwen3.5 in the OpenVINO GenAI explicit modeling framework. This serves as the baseline for future performance optimization work.

Status: K+1 Verify — All Configs Quality Verified   K+1 verify speculative decoding is fully implemented and benchmarked on Qwen3.5-9B (Intel Arc Pro 140T). INT4 FC quality issue resolved via OV_GPU_USE_ONEDNN=0 (INT4 only) + OV_GPU_SDPA_SINGLE_TOKEN_THRESHOLD (INT4 only). INT8 batch performance fixed: oneDNN/SDPA workarounds are now quant-conditional (INT4-only), restoring INT8 batch mode from 1.79 tok/s to 6.87 tok/s (3.8× speedup). All TEXT and VL configs produce coherent text (QUALITY: OK). Per-step profiling instrumentation added (OV_GENAI_STEP_PROFILE=1). VL mode K=2 achieves +15.6% throughput; TEXT mode shows no MTP benefit on this model/hardware.

Speculative tokens: Configurable via --mtp-k N (default: 1). The MTP model drafts K tokens autoregressively using its own hidden_states output. The main model verifies all K drafts in a single K+1 token forward pass.

Architecture

MTP Model Pipeline

Input: input_ids [B, S], hidden_states [B, S, H]
  ↓
1. mtp.embed_tokens(input_ids)           → [B, S, H]       (shared with main model)
2. mtp.pre_fc_norm_embedding(embed)      → [B, S, H]
   mtp.pre_fc_norm_hidden(hidden)        → [B, S, H]
3. Concat([norm_embed, norm_hidden], -1) → [B, S, 2*H]
4. mtp.fc(concat)                        → [B, S, H]       (linear: 2*H → H)
5. MTP decoder layers (full_attention only, dense MLP, typically 1 layer)
6. mtp.norm(out)                         → [B, S, H]
7. lm_head(norm)                         → [B, S, V]       (shared with main model)

Speculative Decode Loop (v1 — 2-Token Verify, replaced)

Given: next_id (confirmed token), mtp_draft_id (MTP's prediction)

1. Feed [next_id, draft_id] as 2-token sequence to main model
2. Main model outputs logits at both positions:
   - logits[0] → verified_id  (what comes after next_id)
   - logits[1] → bonus_id     (what comes after draft_id)
3. Compare verified_id == draft_id:

   ACCEPT:
   - Emit both verified_id and bonus_id (2 tokens per infer)
   - past_len += 2
   - Run MTP on bonus_id's hidden_states → new draft

   REJECT:
   - Emit only verified_id (1 token per infer)
   - Trim main model KV cache by 1 (remove draft's position)
   - Reset MTP state (KV cache invalid after rejection)
   - Run MTP on verified_id's hidden_states → new draft

Speculative Decode Loop (v2 — 1-Token Verify, superseded by v3)

Given: next_id (confirmed token), mtp_draft_id (MTP's prediction)

1. Main model 1-token infer(next_id) → logits, hidden_states
2. Sample verified_id from logits
3. Compare verified_id == mtp_draft_id:

   HIT:
   - Emit verified_id. KV cache already correct (no trim needed).
   - Main model 1-token infer(verified_id) → logits_B, hs_B
   - Sample bonus_id from logits_B, emit bonus_id
   - MTP draft from hs_B → new draft_id
   - decode_steps += 2, past_len += 2 (2 main infers, 0 KV trims)

   MISS:
   - Emit verified_id (no KV trim needed)
   - MTP draft from hidden_states → new draft_id
   - decode_steps += 1, past_len += 1 (1 main infer, 0 KV trims)
Problem: Tok/Infer = 1.00 always (HIT: 2 infers/2 tokens, MISS: 1 infer/1 token). MTP never reduces main model calls — it is pure overhead.

Speculative Decode Loop (v3 — 2-Token Batch Verify, current)

Given: next_id (confirmed token), mtp_draft_id (MTP's prediction)

1. Feed [next_id, draft_id] as 2-token input to main model
2. Main model outputs logits at both positions:
   - logits[0] → verified_id  (what comes after next_id)
   - logits[1] → bonus_id     (what comes after draft_id)
3. Compare verified_id == draft_id:

   HIT:
   - Emit verified_id + bonus_id (2 tokens from 1 infer)
   - KV cache correct, no trim. past_len += 2
   - MTP draft from hs[1] → new draft_id
   - Tok/Infer = 2.0

   MISS:
   - Trim KV cache by 1 (remove draft position). past_len += 1
   - Emit verified_id only (1 token from 1 infer)
   - Reset MTP state, MTP draft from hs[0] → new draft_id
   - Tok/Infer = 1.0

E[Tok/Infer] = h × 2 + (1−h) × 1 = 1 + h
Key improvement over v2: HIT actually skips a main model call (2 tokens from 1 infer). On memory-bound models (9B+), T2tok ≈ T1tok, so 2-token input has near-zero overhead.

Key Implementation Decisions

  1. Fresh tensors per speculative step: 2-token input tensors are created fresh each iteration rather than pre-allocated, to avoid GPU plugin output reallocation issues when switching between seq_len=1 and seq_len=2.
  2. MTP state reset on rejection: On draft rejection, MTP KV cache is fully reset (mtp_request->reset_state()) rather than trimmed. GPU plugin crashes when set_state() is called with CPU tensors on the MTP model after trim. Main model KV trim works correctly.
  3. Output copy before input reset: After 2-token inference, logits and hidden states are copied to local tensors before resetting inputs back to 1-token shapes, because GPU plugin may invalidate tensor references on input change.
  4. Shared weights: embed_tokens and lm_head weights are shared between main model and MTP model via PackedMapping weight loading rules.

GPU Plugin Behaviors Discovered

  • state.get_state() triggers GPU→CPU copy (~0.1 ms per state × 48+ states = significant)
  • state.set_state() triggers CPU→GPU copy (same cost)
  • ROI tensor views (ov::Tensor(old, begin, end)) passed to set_state() do not avoid the copy — GPU plugin still does an internal roundtrip. Tested and confirmed: ROI trim was actually slower (252 ms vs 127 ms)
  • Setting tensors with different shapes triggers GPU recompilation/reallocation
  • Reading output tensors after set_tensor with new input shapes causes ~418 ms GPU sync stall
  • Attention mask: build_kv_causal_mask_with_attention_from_q_len derives KV length from attention_mask dim, stale KV entries will be attended to — trim is required for correctness in 2-tok approach

Break-Even Analysis

For v2 (1-token verify) to match the baseline of 14.3 ms/token:

ms/token = [h × 33.7 + (1−h) × 19.4] / (1 + h) = 14.3
  ⇒ 19.4 + 14.3h = 14.3 + 14.3h
  ⇒ 19.4 = 14.3  (never true)

At the current latency profile (T1=14.3 ms, TMTP=3.4 ms), the MTP overhead (TMTP + logit extract) on every attempt guarantees ms/token > baseline. The overhead floor with 100% hit rate would be:

At h=100%: ms/token = 33.7 / 2 = 16.85 ms/token  (vs 14.3 baseline = 18% overhead)

This is because the MTP model itself adds ~5.1 ms of overhead per token-pair on the HIT path (TMTP + extract), which can never be amortized away when T1 is already very fast (14.3 ms).

For v1 (2-token verify), the break-even requires:

  [h × 29 + (1−h) × 35] / (1 + h) = 14.3
  ⇒ 35 − 6h = 14.3 + 14.3h
  ⇒ h = 20.7 / 20.3 ≈ 102%  (impossible)

Neither approach can break even on this 0.8B model because the base infer latency is too low (14.3 ms) for the fixed overhead costs (MTP infer ~3.4 ms, KV trim ~5 ms) to be amortized. However, with larger models where T1 >> TMTP, the overhead ratio shrinks and MTP becomes profitable.

Draft Token Verification: Algorithm Comparison

v1: 2-Token Verify

Feed confirmed token + MTP draft token together to the main model in a single 2-token forward pass:

Input to main:  [confirmed_tok, draft_tok]     ← 2 tokens
Output:         logits[0] → verified_id    (what comes after confirmed_tok)
                logits[1] → bonus_id       (what comes after draft_tok)

Compare verified_id vs draft_tok:
  ACCEPT (match):    emit verified_id + bonus_id → 2 tokens/step
                     past_len += 2
                     run MTP on bonus_id’s hidden_states → new draft

  REJECT (mismatch): emit verified_id only → 1 token/step
                     trim KV cache by 1 (remove draft position)
                     reset MTP state
                     run MTP on verified_id’s hidden_states → new draft
Problems:
• 2-token infer on 0.8B: ~25 ms (1.75× baseline 14.3 ms)
• Reject requires KV trim: 4–6 ms GPU→CPU→GPU roundtrip
• MTP state reset discards speculative context

v2: 1-Token Verify

Feed only the confirmed token. Draft token never enters the main model KV cache:

Input to main:  [confirmed_tok]                ← 1 token only
Output:         logits → verified_id

Compare verified_id vs draft_tok:
  MATCH:   draft correct → next step uses draft_tok as confirmed
           MTP already produced next_draft in previous step
           → equivalent to 2 tokens/step (skip one main model call)

  MISMATCH: draft wrong → use verified_id as next confirmed
            NO KV trim needed (draft never entered KV cache)
            re-run MTP → new draft
            → 1 token/step
Core advantages:
• Only 1-token infer per step: 14.3 ms vs 25 ms
• Never needs KV trim: saves 4–6 ms per rejection
• E[tokens/step] still = 1 + h (same as v1)
Problem (exposed by 9B benchmark):
• Tok/Infer = 1.00 always — MTP never reduces main model inference count
• HIT: 2 main infers / 2 tokens = 1.0. MISS: 1 infer / 1 token = 1.0
• MTP becomes pure overhead (MTP infer + sampling cost added to every step)

v3: 2-Token Batch Verify (current implementation)

Combines v1’s 2-token verification with correct hidden_states extraction. HIT produces 2 tokens from 1 main infer:

Input to main:  [next_id, draft_id]            ← 2 tokens as batch
Output:         logits[0] → verified_id        (verify next_id)
                logits[1] → bonus_id           (draft_id position)
                hidden_states[0], [1]

Compare verified_id vs draft_id:
  HIT (match):
    Emit verified_id + bonus_id → 2 tokens
    KV cache correct, no trim. past_len += 2
    MTP draft from hs[1] → new draft
    1 main infer → 2 tokens.  Tok/Infer = 2.0

  MISS (mismatch):
    Trim KV cache by 1 (remove draft position). past_len += 1
    Emit verified_id only → 1 token
    Reset MTP KV. MTP draft from hs[0] → new draft
    1 main infer + trim → 1 token. Tok/Infer = 1.0
Key improvements:
• HIT: Tok/Infer = 2.0 (v2 was always 1.0)
• E[Tok/Infer] = 1 + h (e.g., 1.3 at 30% hit rate)
• On memory-bound 9B+: T2tok ≈ T1tok, so 2-token infer is near-free
• On CPU: KV trim is pure memory copy (no GPU roundtrip), minimal cost

vLLM: Standard Speculative Decoding (K+1 Token Verify)

vLLM uses the Leviathan et al. 2023 algorithm—pack all K draft tokens into a single forward pass:

Confirmed token: tok_c
MTP proposer outputs K drafts: [d1, d2, ..., dk]

Phase 1 — Target Forward (single inference):
  Input:  [tok_c, d1, d2, ..., dk]        ← K+1 tokens
  Output: logits at K+1 positions

Phase 2 — Rejection Sampling (sequential):
  Greedy (temperature=0):
    pos 0: argmax(logits[0]) == d1? → accept/reject
    pos 1: argmax(logits[1]) == d2? → accept/reject (only if prev accepted)
    ...
    Stop at first reject → emit target’s token
    All accept → emit d1..dk + bonus_token (K+1 tokens!)

  Probabilistic (temperature>0):
    accept_prob = min(1, p_target[d_i] / p_draft[d_i])
    reject → sample from max(0, p_target - p_draft)
    Guarantees output distribution = target model exactly

Phase 3 — KV Cache:
  vLLM does NOT trim KV cache
  - All K+1 KV entries remain (PagedAttention)
  - num_rejected_tokens adjusts seq_lens
  - Stale entries overwritten in next forward

Why v3 Tok/Infer = 1 + h (while v2 = 1.00)

v3 (2-token batch verify):
  Each step: 1 main model infer([next_id, draft_id])

  Hit (probability h):
    logits[0] → verified_id == draft_id ✓
    logits[1] → bonus_id
    → 2 tokens, 1 main infer.  Tok/Infer = 2.0

  Miss (probability 1−h):
    logits[0] → verified_id ≠ draft_id
    → 1 token, 1 main infer + trim.  Tok/Infer = 1.0

  E[Tok/Infer] = h × 2 + (1−h) × 1 = 1 + h

v2 (1-token verify) problem:
  Hit: 2 main infers / 2 tokens = Tok/Infer = 1.0
  Miss: 1 main infer / 1 token = Tok/Infer = 1.0
  → Tok/Infer always 1.0, MTP never reduces inference count

Key Insights

1. v3 solves v2’s core problem: v2 Tok/Infer=1.00 means MTP is pure overhead. v3 restores 2-token batch verify so HIT actually skips a main model call.

2. v3 suits memory-bound large models: On 9B+ models, T2tok ≈ T1tok, so 2-token infer has near-zero extra cost.

3. v2 suits compute-bound small models: On 0.8B, T2tok = 1.75× T1tok, making 2-token infer expensive.

4. vLLM K+1 verify excels on large models: When T(K+1)tok ≈ T1tok (memory-bound), verifying extra drafts costs nearly zero.

5. CPU KV trim cost is low: Pure memory copy, unlike GPU’s GPU→CPU→GPU roundtrip (4–6 ms).

Greedy Rejection Algorithm (Triton kernel)

For greedy decoding (temperature=0), the kernel at rejection_greedy_sample_kernel L653:

for pos in range(num_draft_tokens):
    if not rejected:
        draft_token_id = draft_token_ids[start + pos]
        target_argmax_id = target_argmax[start + pos]   # argmax(logits[pos])
        output[req, pos] = target_argmax_id             # always emit target's choice
        if draft_token_id != target_argmax_id:
            rejected = True                             # stop here

if not rejected:
    output[req, num_draft_tokens] = bonus_token_id      # bonus token from logits[K]
Key behavior: Even on REJECT, the output is the target model’s sampled token (not the draft). This means every verify step produces at least 1 correct token. On full accept (all K drafts correct), a bonus token is appended = K+1 tokens per verify.

Random/Probabilistic Rejection Algorithm

For sampling (temperature > 0), at rejection_random_sample_kernel L698:

for pos in range(num_draft_tokens):
    if not rejected:
        draft_prob = p_draft[pos, draft_token_id]
        target_prob = p_target[pos, draft_token_id]
        u = uniform_random()

        if draft_prob > 0 and target_prob / draft_prob >= u:
            output[req, pos] = draft_token_id           # ACCEPT draft
        else:
            output[req, pos] = recovered_token_ids[pos]  # REJECT → recovered token
            rejected = True                              # from max(0, p_target - p_draft)

if not rejected:
    output[req, num_draft_tokens] = bonus_token_id

The recovered token is sampled from the residual distribution max(0, ptarget − pdraft), normalized. This guarantees the output distribution matches the target model exactly (Leviathan et al. 2023).

KV Cache Management on Partial Accept

vLLM uses a no-trim strategy for KV cache on partial acceptance:

  1. During target forward, KV cache is populated for ALL K+1 positions (confirmed + all K drafts)
  2. On rejection at position j (only j of K accepted), the KV entries for positions j+1..K remain in cache
  3. On the next step, num_computed_tokens is set to reflect only accepted positions
  4. Stale KV entries are overwritten when those positions are recomputed with correct tokens
  5. num_rejected_tokens_gpu tensor adjusts seq_lens in the drafter: seq_lens -= num_rejected_tokens
Contrast with our OpenVINO v1 approach: We had to explicitly trim KV cache via GPU→CPU→GPU roundtrip, costing ~4–6 ms per rejection. The OpenVINO GPU plugin does not support overwriting stale KV entries because append_kv_cache always appends. Our v2 (1-token verify) sidesteps this entirely.

Kernel-Level Per-Token State Snapshots (April 2026)

Problem: Linear State Corruption at K≥2

Qwen3.5’s hybrid architecture uses 24 GatedDeltaNet recurrent linear attention layers (every layer except every 4th, which is full SDPA). Each recurrent state evolves as: 

S_t = exp(g_t) · S_{t-1} + k_t ⊗ β_t(v_t − S_{t-1}^T k_t)

This recurrence is irreversible: given S_t, you cannot recover S_{t-1}. When MTP batch verify processes K+1 tokens but only num_accepted < K are correct, the rejected tokens have already corrupted the recurrent state. Previous approaches (virtual trim, periodic refresh) only addressed KV cache rollback, leaving the recurrent state “drifting” from accumulated rejected tokens, causing text degeneration at K≥2.

Solution: vLLM-Inspired Per-Token State Snapshots

Following vLLM’s approach for Qwen3.5, we modified the GPU LinearAttention OCL kernel to write per-token intermediate recurrent states to a 3rd output buffer. After batch verification, the host selects the correct state at all_states[:, num_accepted, :, :, :] and writes it back to the variable, achieving zero-drift, zero-re-forward state correction.

GPU-Side State Restore API (April 2026)

To avoid the CPU round-trip overhead of get_tensor() + memcpy + set_state(), we added a new restore_variable_from_output() API through the full OpenVINO inference stack. This copies directly from GPU output memory to variable state memory using enqueue_memcpy, with no host-side data transfer.

Pre-Reorder Memory Optimization

The GPU plugin inserts an f16→f32 reorder primitive at every graph output boundary (the model is declared in FP32, but the GPU computes internally in FP16). When restore_variable_from_output() detects a type mismatch between the output memory (f32) and the variable state (f16), it navigates to the reorder’s input dependency via network->get_primitive(name)->dep_memory_ptr(0) to obtain the pre-reorder f16 memory. This enables a direct GPU-to-GPU enqueue_memcpy with no type conversion and no CPU involvement.

    Root Cause: oneDNN INT4 FC Batch-Size-Dependent Divergence

    On the Intel Arc Pro 140T (IMMAD hardware), the GPU plugin uses oneDNN for fully-connected (FC) layers rather than the OCL bf_tiled kernel. The oneDNN INT4 GEMM produces different numerical results for batch_size=1 vs batch_size=2, causing:

    • MTP draft phase (batch=1): produces logits X
    • Main verify phase (batch=2): produces logits X′ (numerically different for same token position)
    • The divergence compounds across speculative decode steps, causing text degeneration
    • TEXT mode is less affected because text prompts produce less extreme activation ranges

    Why GPU_USE_ONEDNN Cannot Be Set via API

    The property GPU_USE_ONEDNN has RELEASE_INTERNAL visibility in the GPU plugin, meaning it cannot be passed through ov::Core::compile_model() properties. It can only be set via the environment variable OV_GPU_USE_ONEDNN=0.

    Fix: Disable oneDNN + Force OCL bf_tiled for Small Batches

    # Disable oneDNN for FC layers:
set OV_GPU_USE_ONEDNN=0

# Force OCL bf_tiled kernel tile_b=1 for batch<=2:
set OV_GPU_FC_SINGLE_BATCH_THRESHOLD=2

    With both env vars set, the GPU plugin uses the OCL bf_tiled kernel for all FC layers. The FC_SINGLE_BATCH_THRESHOLD=2 ensures that batch=1 and batch=2 both use the tile_b=1 code path, producing identical numerical results for the same token position regardless of batch size.

    Verification

    With OV_GPU_USE_ONEDNN=0 + OV_GPU_FC_SINGLE_BATCH_THRESHOLD=2: all TEXT and VL configs (K=1, K=2, K=3) produce coherent, quality-verified text. Benchmark script bench_mtp.ps1 reports QUALITY: OK for all configurations.

    Performance Impact

    Disabling oneDNN incurs a performance cost of approximately 24% regression in raw throughput compared to the oneDNN path. This is a known trade-off: correctness over performance. A proper fix would require modifying the oneDNN INT4 GEMM kernel to produce batch-size-invariant results.

    vLLM vs OpenVINO: State Management for Hybrid Models

    Qwen3.5’s hybrid architecture has both Transformer attention layers and GatedDeltaNet linear attention (Mamba-like) layers. The linear attention layers maintain internal recurrent state (linear_states + conv_states) that must be correctly managed during speculative decoding. When draft tokens are rejected, the states updated by rejected tokens must be rolled back.

    State Rollback: Two Approaches

    AspectOpenVINO (our impl)vLLM (Qwen3Next)
    StrategySnapshot → Verify → RestoreBlock-based multi-slot (write-ahead)
    On rejectionGPU memcpy restore from snapshot bufferRead from block[last_accepted_idx] — no copy needed
    On acceptanceUpdate snapshotCopy state to running block (batch memcpy)
    Memory overhead1× snapshot buffer (48 linear + 24 conv tensors)num_spec_blocks × state per request
    Restore costO(state_size) per rejection (~5.7 ms, 3.1% of decode)≈0 (state already in correct block)
    Drift riskNone (GPU-side exact restore)None (state always in correct block)
    Periodic refreshNot needed for K≤2 with OCL FCNot needed
    ImplementationGPU plugin restore_variable_from_output()Triton batch_memcpy kernel

    vLLM Block-Based Multi-Slot Design

    vLLM’s approach (in vllm/v1/worker/mamba_utils.py) avoids explicit snapshot/restore entirely:

    1. Pre-allocate extra block slots: Each request gets num_speculative_blocks extra state blocks beyond what’s needed for real tokens.
    2. Write state as tokens are processed: Each speculative token’s Mamba state is written into its own block slot via ssm_state_indices[batch, token_idx].
    3. On next iteration: preprocess_mamba() reads from block[num_accepted-1] as the initial state — effectively “rolling back” for free by choosing the right read address.
    4. Block boundary crossings: A single batched batch_memcpy Triton kernel handles all layers and all requests when state crosses block boundaries.

    Key files: vllm/model_executor/models/qwen3_next.py (implements HasInnerState), vllm/model_executor/layers/mamba/gdn_linear_attn.py (GatedDeltaNet with spec decode support), vllm/model_executor/layers/fla/ops/fused_recurrent.py (Triton kernel with IS_SPEC_DECODING flag).

    Why Our 3.1% Restore Overhead Is Acceptable

    While vLLM’s block-based design is theoretically zero-cost for state rollback, our snapshot/restore approach has practical advantages in the OpenVINO GPU plugin context:
    • 3.1% overhead is modest: Restore costs 5.67 ms/step vs 280.96 ms for the main infer — a 50× difference.
    • No kernel modifications needed: The block-based approach requires modifying every Mamba/GDN kernel to accept per-token state indices. Our approach works with the existing GPU plugin restore_variable_from_output() API.
    • The dominant bottleneck is elsewhere: Main verify (80%) and MTP draft (20%) determine throughput. Eliminating restore entirely would yield <4% improvement.
