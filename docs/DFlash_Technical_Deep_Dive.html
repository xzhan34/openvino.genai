1. 概述：什么是 DFlash？

一句话总结：DFlash 是一种基于块扩散模型（Block Diffusion）的投机解码框架， 使用轻量级扩散草稿模型（drafter）在单次前向传播中并行生成多个候选 token， 再由目标大模型并行验证，实现无损加速超过 6×，比 EAGLE-3 快最多 2.5×。

🎯 核心定位

将扩散语言模型（dLLM）定位为投机解码的专用起草器， 而非完整的生成模型。利用扩散的并行性消除顺序起草瓶颈。

  • 通过 KV 注入将条件信息持久注入每一层
  • 单次前向并行生成整个 token 块

2. 背景：为什么需要 DFlash？

2.1 自回归解码的瓶颈

大语言模型（LLM）的推理本质是逐 token 顺序生成，每个 token 必须等待之前所有 token 计算完成。 这导致：

  • 推理延迟高：受限于顺序依赖，无法充分利用 GPU 并行能力
  • GPU 利用率低：单 token 生成时计算/内存比极低（memory-bound）
  • 长 CoT 推理更严重：推理模型（如 o1、DeepSeek-R1）生成链长达数千 token

2.2 现有投机解码的局限

投机解码（Speculative Decoding）的基本范式是：

小草稿模型
生成 γ 个候选 token
→
大目标模型
并行验证
→
接受匹配的 token
拒绝不匹配的

但现有方法（如 EAGLE 系列、Medusa）的草稿模型仍然自回归生成，存在两个根本限制：

限制 1 起草延迟线性增长

自回归起草 γ 个 token 需要 γ 次前向传播：

Tdraft = γ · tstep

为了控制延迟，被迫使用极浅的模型（如 EAGLE-3 仅 1 层 Transformer），严重限制了草稿质量。

限制 2 错误累积

自回归起草中，前面 token 的错误会级联传播到后续所有 token，导致接受长度快速饱和。 实际加速比被限制在 2-3× 左右。

2.3 扩散 LLM 的机遇与挑战

扩散语言模型（如 LLaDA、Block Diffusion）可以并行生成多个 token，但作为独立生成模型使用时：

  • 生成质量通常不如自回归模型
  • 需要多步去噪才能保证质量，抵消了并行优势
  • 大型扩散起草器（如 7B 参数的 DiffuSpec）内存开销过大
DFlash 的关键洞察： 不要让扩散模型独立工作——让它作为目标模型的"扩散适配器"， 借助目标模型的隐藏特征来获得高质量的上下文信息，从而实现又快又准的并行起草。

3. 核心思想：扩散模型 × 投机解码

DFlash 的核心洞察可以总结为一句话："the target knows best"（目标模型最了解答案）。

为什么这个思路有效？

大型自回归 LLM 的隐藏层特征不仅编码了当前 token 的信息，还隐式包含了关于未来多个 token 的预测信息 （参见 Samragh et al., 2025）。DFlash 利用这些特征作为条件，让草稿模型不需要"从头推理"， 而是在目标模型的"指导"下进行并行预测。

这创造了一个双赢组合：

组件 角色 优势
大型自回归目标模型 提供上下文特征 + 验证 高质量推理能力，保证输出无损
轻量扩散草稿模型 并行起草 token 块 单次前向传播完成，延迟极低

4. 系统架构与推理流程

4.1 整体推理流程

DFlash 的推理分为两个阶段：

阶段 1：预填充（Prefill）

输入 prompt
→
目标模型前向传播
→
生成第 1 个 token
+ 提取隐藏特征

在 prefill 过程中，从目标模型的均匀采样的固定层（如第 2 层到倒数第 3 层之间的 5 个层） 提取隐藏状态（hidden states），然后通过一个轻量投影层（Linear + RMSNorm）将多层特征拼接融合为 紧凑的目标上下文特征。

阶段 2：投机解码循环

1. 用 mask token
填充块位置
→
2. 草稿模型
单次并行预测
→
3. 目标模型
并行验证
→
4. 确定接受长度
保留正确 token

每个循环中：

  1. 当前位置开始，填充 block_size 个位置（第一个为已知的 bonus token，其余为 mask token）
  2. 草稿模型接收 noise embedding（mask token 的嵌入）和目标上下文特征，一次前向传播预测所有位置
  3. 目标模型对草稿结果进行并行验证
  4. 逐位置比较，找到第一个不匹配的位置，之前的全部接受，不匹配位置使用目标模型的输出作为 bonus token
  5. 更新上下文特征，进入下一个循环

4.2 草稿模型架构

DFlash 的草稿模型（DFlashDraftModel）是一个改造过的 Qwen3 Transformer：

组件说明
层数5 层（默认）或 8 层（Coder）— 比 EAGLE-3 的 1 层深得多
隐藏维度与目标模型相同（如 Qwen3-8B 的 4096）
注意力自定义 Qwen3DFlashAttention，支持 KV 注入
Embedding 层与目标模型共享，冻结不训练
LM Head与目标模型共享，冻结不训练
投影层fc：将多层隐藏特征投影到 hidden_size；hidden_norm：RMSNorm 归一化
block_size16（默认）或 10（LLaMA）

5. 关键技术 1：KV 注入条件化

这是 DFlash 最核心的设计创新，也是其高接受率的关键原因。

5.1 与 EAGLE 的对比

EAGLE-3 方式

将目标模型的隐藏特征与草稿模型的 token embedding 在输入端融合， 仅作为草稿模型的输入。随着层数增加，目标信息逐层稀释， 导致增加草稿层数对接受长度的提升效果递减。

DFlash 方式 (KV 注入)

将融合后的目标上下文特征视为持久的上下文信息， 直接注入到草稿模型每一层的 Key 和 Value 投影中。 目标信息在所有层中始终保持强条件化，使接受长度能随层数有效扩展。

5.2 KV 注入的实现细节

在源码 Qwen3DFlashAttention.forward() 中，注入逻辑如下：

# 目标上下文特征生成 K, V
k_ctx = self.k_proj(target_hidden)    # shape: [bsz, ctx_len, kv_heads * head_dim]
v_ctx = self.v_proj(target_hidden)

# 草稿 token 自身的 K, V
k_noise = self.k_proj(hidden_states)  # shape: [bsz, q_len, kv_heads * head_dim]
v_noise = self.v_proj(hidden_states)

# 拼接：目标上下文 K/V 在前，草稿 K/V 在后
k = torch.cat([k_ctx, k_noise], dim=1)  # [bsz, ctx_len + q_len, ...]
v = torch.cat([v_ctx, v_noise], dim=1)

# Query 仅来自草稿 token
q = self.q_proj(hidden_states)          # [bsz, q_len, ...]

这意味着草稿模型中每一层的每一个注意力头都能直接 attend 到目标模型的上下文特征， 确保了条件信号不会被稀释。

5.3 上下文特征提取流程

# 从目标模型的指定层提取隐藏状态
def extract_context_feature(hidden_states, layer_ids):
    offset = 1  # 跳过 embedding 层
    selected = [hidden_states[layer_id + offset] for layer_id in layer_ids]
    return torch.cat(selected, dim=-1)  # 拼接所有层的特征

# 通过投影层融合
# fc: Linear(num_layers * hidden_size → hidden_size)
# hidden_norm: RMSNorm
target_hidden = self.hidden_norm(self.fc(target_hidden))

例如，对于 Qwen3-8B（32 层，hidden_size=4096），如果从 5 个层提取特征：

  • 输入维度：5 × 4096 = 20480
  • 经 fc 投影后：4096
  • 经 hidden_norm 归一化后注入每层

6. 关键技术 2：并行扩散起草

6.1 扩散 vs 自回归的起草开销

这是 DFlash 相比 EAGLE 的另一个关键优势：

方式 起草开销公式 γ=16 时的效果
自回归（EAGLE-3, 1 层） Tdraft = γ × tstep 16 次顺序前向传播
扩散（DFlash, 5 层） Tdraft = tparallel 1 次并行前向传播
关键优势： 因为起草开销不随 γ 线性增长，DFlash 可以使用更深（5 层甚至 8 层）的草稿模型， 获得更高的草稿质量和接受长度，同时总延迟仍然更低。 实验表明，5 层 DFlash 生成 16 个 token 比 1 层 EAGLE-3 生成 8 个 token 还快。

6.2 块级扩散过程

在推理时，DFlash 使用以下方式生成一个块：

  1. 已知位置 0 的 token（上一轮验证产生的 bonus token）
  2. 位置 1 到 block_size-1 填充为 mask_token_id
  3. 将整个块通过目标模型的 embed_tokens 获得 noise embedding
  4. 草稿模型接收 noise embedding + 目标上下文特征，一次前向传播同时预测所有 mask 位置
  5. 通过共享的 lm_head 将隐藏状态映射为 logits，采样得到草稿 token

源码中的核心推理循环：

# 草稿模型单次并行预测
draft_logits = target.lm_head(self(
    target_hidden=target_hidden,
    noise_embedding=noise_embedding,
    position_ids=...,
    past_key_values=past_key_values_draft,
    use_cache=True,
    is_causal=False,  # 非因果注意力！块内双向
))
block_output_ids[:, 1:] = sample(draft_logits)

注意 is_causal=False：块内使用双向注意力，允许每个位置看到整个块的信息。

7. 接受率与验证机制

7.1 投机解码的加速公式

投机解码的平均每 token 延迟为：

L = (Tdraft + Tverify) / τ

其中 τ ∈ [1, γ+1] 是每轮循环中期望接受的 token 数量（包括目标模型生成的 bonus token），加速比为：

η = Ltarget / L = Ltarget × τ / (Tdraft + Tverify)

提升加速比有两条路径：增大 τ（接受长度）或减小 Tdraft（起草开销）。 DFlash 在两条路径上都有优势。

7.2 验证与接受长度计算

目标模型对草稿块进行并行验证的过程：

# 目标模型并行验证整个草稿块
output = target(
    block_output_ids,
    position_ids=block_position_ids,
    past_key_values=past_key_values_target,
    use_cache=True,
    output_hidden_states=True,
)

# 计算接受长度：从第一个位置开始，连续匹配的 token 数量
posterior = sample(output.logits, temperature)
acceptance_length = (block_output_ids[:, 1:] == posterior[:, :-1]).cumprod(dim=1).sum(dim=1)[0].item()

# 接受匹配的 token + 使用目标模型在不匹配位置的输出作为 bonus token
output_ids[:, start : start + acceptance_length + 1] = block_output_ids[:, : acceptance_length + 1]
output_ids[:, start + acceptance_length + 1] = posterior[:, acceptance_length]
start += acceptance_length + 1  # 前进 acceptance_length + 1 个 token

验证逻辑的关键：

  • block_output_ids[:, 1:]：草稿模型预测的 token（位置 1 到 block_size-1）
  • posterior[:, :-1]：目标模型对同样位置的预测
  • .cumprod(dim=1)：累积乘积保证了一旦某个位置不匹配，后续所有位置都被拒绝
  • 即使所有草稿 token 被拒绝，目标模型仍然贡献 1 个 bonus token，所以 τ ≥ 1

8. 训练方法

8.1 训练总览

DFlash 的训练是让草稿模型对齐目标模型的输出，具体修改了标准块扩散训练的几个关键环节：

训练设置

  • 目标模型：冻结，用于提取隐藏特征和提供训练标签
  • 可训练参数：仅草稿 Transformer 层 + 投影层（fc + hidden_norm）
  • 共享冻结：token embedding 和 LM head 与目标模型共享且冻结
  • 优化器：AdamW，学习率 6×10⁻⁴，余弦调度，warmup 4%
  • 训练数据：~800K 样本，来自 NVIDIA Nemotron + CodeAlpaca，使用目标模型生成的响应
  • 训练轮数：6 个 epoch

8.2 随机锚点采样构建块

标准块扩散训练将响应均匀分割为块，但 DFlash 改用随机锚点采样：

  • 从响应中随机采样锚点 token
  • 每个锚点作为块的第一个位置（已知的 clean token）
  • 锚点后的 block_size - 1 个位置标记为 mask
  • 草稿模型训练预测这些 mask 位置

这直接模拟了推理时的行为（验证后的 bonus token 作为下一轮的第一个 token）， 并提供更多样化的训练数据。实验表明，这比标准块分割方式显著提高接受长度（5.64 vs 4.94）。

8.3 位置衰减损失权重

在投机解码中，块内靠前位置的错误会使后续所有位置无效。 因此 DFlash 使用指数衰减权重强调早期位置：

wk = exp(−(k−1) / γ)

其中 k 是块内位置，γ 控制衰减速率（block_size=16 时 γ=7）。 这使得训练更快收敛，且最终接受长度更高。

8.4 高效训练设计

  • Flex Attention：所有块拼接为单个序列，使用稀疏注意力掩码：块内双向注意力，块间禁止注意力
  • 离线缓存：支持预计算目标隐藏特征并缓存到磁盘，减少训练时 GPU 开销
  • 长上下文训练：固定每个序列的 mask 块数量，每个 epoch 随机采样锚点位置，提供数据增强

10.2 核心类和函数

DFlashDraftModel — 草稿模型主类

  • 继承自 Qwen3PreTrainedModel
  • forward()：接收 noise_embedding（mask token 嵌入）和 target_hidden（目标特征）， 通过投影融合后逐层传播，返回归一化后的隐藏状态
  • spec_generate()：完整的投机解码推理循环，包含 prefill、起草、验证、接受判定
  • 关键属性：fc（特征投影）、hidden_norm（RMSNorm）、target_layer_ids（要提取的目标层 ID）、 block_size、mask_token_id

Qwen3DFlashAttention — 带 KV 注入的注意力层

  • 核心改动：forward() 接收额外的 target_hidden 参数
  • Q 仅来自草稿 token：q = q_proj(hidden_states)
  • K、V 由目标特征和草稿特征拼接组成
  • 使用与目标模型相同的 QKNorm 和 RoPE

辅助函数

  • build_target_layer_ids(num_target_layers, num_draft_layers)： 均匀选择目标模型中从第 2 层到倒数第 3 层之间的层 ID
  • extract_context_feature(hidden_states, layer_ids)： 从目标模型输出中提取并拼接指定层的隐藏状态
  • sample(logits, temperature)： 温度为 0 时 argmax，否则 softmax + multinomial 采样

10.3 推理循环关键代码注释

def spec_generate(self, target, input_ids, max_new_tokens, stop_token_ids, temperature):
    # === 1. 初始化 ===
    output_ids = torch.full((1, max_length + block_size), mask_token_id)  # 全填 mask
    
    # === 2. Prefill ===
    output = target(input_ids, ..., output_hidden_states=True)           # 目标模型 prefill
    output_ids[:, num_input:num_input+1] = sample(output.logits, temp)   # 生成第 1 个 token
    target_hidden = extract_context_feature(output.hidden_states, ...)   # 提取上下文特征
    
    # === 3. 解码循环 ===
    while start < max_length:
        # 3a. 获取当前块（第一个 token 已知，其余为 mask）
        block_output_ids = output_ids[:, start:start+block_size].clone()
        noise_embedding = target.model.embed_tokens(block_output_ids)
        
        # 3b. 草稿模型并行预测 (单次前向)
        draft_hidden = self(target_hidden=target_hidden, noise_embedding=noise_embedding, ...)
        draft_logits = target.lm_head(draft_hidden[:, -block_size+1:, :])
        block_output_ids[:, 1:] = sample(draft_logits)
        
        # 3c. 目标模型并行验证
        output = target(block_output_ids, ..., output_hidden_states=True)
        posterior = sample(output.logits, temperature)
        
        # 3d. 计算接受长度
        match = (block_output_ids[:, 1:] == posterior[:, :-1])  # 逐位置比较
        acceptance_length = match.cumprod(dim=1).sum(dim=1)[0]  # 连续匹配数
        
        # 3e. 接受 + bonus token
        output_ids[:, start:start+acceptance_length+1] = block_output_ids[:, :acceptance_length+1]
        output_ids[:, start+acceptance_length+1] = posterior[:, acceptance_length]  # bonus
        start += acceptance_length + 1
        
        # 3f. 更新上下文特征
        target_hidden = extract_context_feature(output.hidden_states, ...)[:, :acceptance_length+1, :]

3.2 apply_rotary_pos_emb() — 旋转位置编码

def apply_rotary_pos_emb(q, k, cos, sin, position_ids=None, unsqueeze_dim=1):
    cos = cos.unsqueeze(unsqueeze_dim)
    sin = sin.unsqueeze(unsqueeze_dim)
    q_len = q.size(-2)
    # 关键：Q 只取最后 q_len 个位置的 cos/sin
    q_embed = (q * cos[..., -q_len:, :]) + (rotate_half(q) * sin[..., -q_len:, :])
    k_embed = (k * cos) + (rotate_half(k) * sin)
    return q_embed, k_embed

此函数与旧版完全相同。核心特点：

  • Q 取最后 q_len 个位置：因为 DFlash 的注意力机制中 K/V 包含 context + noise 两部分（长度为 ctx_len + q_len），而 Q 只是 noise 部分（长度 q_len），所以 Q 的 RoPE 只需要对应最后 q_len 个位置
  • K 使用全部位置：K 需要覆盖 context 和 noise 的完整位置序列
  • 使用标准的 rotate_half 实现（从 Qwen3 导入的），将向量分为两半并旋转

3.3 Qwen3DFlashAttention — 核心注意力机制

3.3.1 初始化

class Qwen3DFlashAttention(nn.Module):
    def __init__(self, config: Qwen3Config, layer_idx: int):
        # 标准 Qwen3 注意力参数
        self.head_dim = getattr(config, "head_dim", config.hidden_size // config.num_attention_heads)
        self.num_key_value_groups = config.num_attention_heads // config.num_key_value_heads  # = 32/8 = 4
        self.scaling = self.head_dim ** -0.5      # = 128^(-0.5) ≈ 0.0884
        self.is_causal = False                     # ⚡ 关键：非因果注意力！

        # QKV + Output 投影
        self.q_proj = nn.Linear(2560, 32 * 128, bias=False)   # [2560 → 4096]
        self.k_proj = nn.Linear(2560,  8 * 128, bias=False)   # [2560 → 1024] (GQA)
        self.v_proj = nn.Linear(2560,  8 * 128, bias=False)   # [2560 → 1024] (GQA)
        self.o_proj = nn.Linear(4096, 2560, bias=False)       # [4096 → 2560]

        # QK 归一化（Qwen3 特有）
        self.q_norm = Qwen3RMSNorm(128, eps=1e-6)
        self.k_norm = Qwen3RMSNorm(128, eps=1e-6)

        # 滑动窗口（当 layer_types 指定时启用）
        self.sliding_window = config.sliding_window \
            if config.layer_types[layer_idx] == "sliding_attention" else None

重要设计决策：

  • is_causal = False — DFlash 使用双向注意力（non-causal），因为 draft token 之间是并行生成的扩散过程，不是从左到右的自回归序列
  • GQA (Grouped Query Attention) 比例 4:1 — 每 4 个 query head 共享 1 组 KV head，与目标模型 Qwen3.5-4B 相同
  • QK Norm — 在 RoPE 之前对 Q、K 分别施加 RMSNorm（Qwen3 架构特色，不同于标准 LLaMA）
  • sliding_window — 支持滑动窗口注意力（当前配置均为 full_attention，但代码已预留支持）

3.3.2 Forward — KV 注入机制

def forward(self, hidden_states, target_hidden, position_embeddings, attention_mask, ...):
    bsz, q_len = hidden_states.shape[:-1]     # noise tokens 数量
    ctx_len = target_hidden.shape[1]           # context tokens 数量

    # Step 1: Q 仅来自 noise（draft tokens）
    q = self.q_proj(hidden_states)              # [B, q_len, 4096]
    q = self.q_norm(q.view(bsz, q_len, -1, self.head_dim)).transpose(1, 2)

    # Step 2: K/V 拼接 context + noise
    k_ctx   = self.k_proj(target_hidden)        # [B, ctx_len, 1024]
    k_noise = self.k_proj(hidden_states)        # [B, q_len,  1024]
    v_ctx   = self.v_proj(target_hidden)
    v_noise = self.v_proj(hidden_states)
    k = torch.cat([k_ctx, k_noise], dim=1)     # [B, ctx_len+q_len, 1024]
    v = torch.cat([v_ctx, v_noise], dim=1)

    # Step 3: K norm + RoPE
    k = self.k_norm(k.view(...)).transpose(1, 2)
    cos, sin = position_embeddings
    q, k = apply_rotary_pos_emb(q, k, cos, sin)

    # Step 4: KV Cache (if present)
    if past_key_values is not None:
        k, v = past_key_values.update(k, v, self.layer_idx, ...)

    # Step 5: 执行注意力计算
    attn_output, _ = attn_fn(self, q, k, v, attention_mask, ...)
DFlash KV 注入流程图：

target_hidden (从目标模型抽取) ─── K_proj/V_proj ───→ k_ctx, v_ctx
⬇ concat
hidden_states (noise embedding) ─── K_proj/V_proj ───→ k_noise, v_noise

K = [k_ctx; k_noise]    V = [v_ctx; v_noise]
⬇
Q (from noise only) ──── Attention(Q, K, V) ────→ output

这是 DFlash 的核心创新机制——通过将目标模型的中间层 hidden states 注入到 draft 模型的 KV 中，draft 模型能够以极少参数量获得目标模型的知识引导。两个版本的 KV 注入逻辑完全相同。

3.4 Qwen3DFlashDecoderLayer — Transformer 解码层

class Qwen3DFlashDecoderLayer(GradientCheckpointingLayer):
    def __init__(self, config, layer_idx):
        self.self_attn = Qwen3DFlashAttention(config, layer_idx)
        self.mlp = Qwen3MLP(config)                     # SwiGLU MLP
        self.input_layernorm = Qwen3RMSNorm(2560, eps=1e-6)
        self.post_attention_layernorm = Qwen3RMSNorm(2560, eps=1e-6)

    def forward(self, target_hidden, hidden_states, ...):
        # Pre-norm → Attention → Residual
        residual = hidden_states
        hidden_states = self.input_layernorm(hidden_states)
        hidden_states = self.self_attn(
            hidden_states=hidden_states,
            target_hidden=target_hidden,  # ← 注入目标特征
            ...
        )[0]
        hidden_states = residual + hidden_states

        # Pre-norm → MLP → Residual
        residual = hidden_states
        hidden_states = self.post_attention_layernorm(hidden_states)
        hidden_states = self.mlp(hidden_states)
        hidden_states = residual + hidden_states
        return hidden_states

标准的 Pre-LayerNorm Transformer 解码器块，与旧版完全相同。每层都接收 target_hidden 参数并传递给注意力模块。

MLP 使用 Qwen3 的 SwiGLU 架构：gate_proj (2560→9728) + up_proj (2560→9728) → SiLU gate → down_proj (9728→2560)

def forward(self, position_ids, attention_mask=None,
             noise_embedding=None, target_hidden=None,
             past_key_values=None, use_cache=False, **kwargs):

def build_target_layer_ids(num_target_layers, num_draft_layers):
    if num_draft_layers == 1:
        return [(num_target_layers // 2)]   # 单层：取中间层
    start = 1
    end = num_target_layers - 3
    span = end - start

        for i in range(num_draft_layers)

    return target_layer_ids

    selected = [hidden_states[lid + offset] for lid in layer_ids]
    return torch.cat(selected, dim=-1)

从目标模型的所有层输出中，按 layer_ids 选取特定层的 hidden states 并沿最后维度拼接。新版中此操作由推理框架在调用 draft model 之前完成。

7. DFlash 推理原理回顾

DFlash (arXiv:2602.06036) 是一种基于块扩散（Block Diffusion）的推测解码方法，核心思想是：

  1. Block Diffusion Draft：将推测解码中的 draft 步骤替换为一个轻量级离散扩散模型。给定一个 block 的 mask tokens，扩散模型在单步去噪中并行预测所有 token
  2. KV 注入：draft 模型不直接接收目标模型的 logits 或 token，而是注入目标模型的中间层隐藏状态（跨多个深度层），作为注意力的 K/V 部分引导生成
  3. 非因果注意力：由于是扩散生成而非自回归，draft tokens 之间使用双向注意力（is_causal=False），每个 draft token 都能看到 block 内其他 token
  4. 目标模型验证：draft 后目标模型对整个 block 执行一次前向传播进行验证，接受匹配的前缀

相比传统 speculative decoding（如 Eagle, Medusa）的优势：

  • 无需自回归 draft（每次只 1-step diffusion），draft 速度更快
  • 通过 KV 注入获得比小型 AR draft model 更强的对齐效果
  • Block size 16 时，一次 draft 可生成 16 个候选 token

DFlash 是什么？

DFlash 是一种新型推测解码（Speculative Decoding）方法，使用轻量级的块扩散（Block Diffusion）模型作为草稿模型（drafter），与大型目标模型（如 Qwen3-4B）配合实现推理加速。

A.3 utils.py — 工具函数

A.3.1 build_target_layer_ids

def build_target_layer_ids(num_target_layers: int, num_draft_layers: int):
    if num_draft_layers == 1:
        return [(num_target_layers // 2)]         # 只1层时取中间层
    start = 1                                    # 从第1层开始
    end = num_target_layers - 3                   # 到倒数第3层（36-3=33）
    span = end - start                              # 有效范围 = 32
    target_layer_ids = [
        int(round(start + (i * span) / (num_draft_layers - 1)))
        for i in range(num_draft_layers)
    ]
    return target_layer_ids

算法：在 [1, num_target_layers-3] 范围内均匀采样 num_draft_layers 个点。跳过第 0 层（太浅）和最后 2 层（太特化），均匀分布捕获不同抽象层级的特征。

A.3.2 extract_context_feature

def extract_context_feature(
    hidden_states: list[torch.Tensor],   # target 模型输出的所有层 hidden_states
    layer_ids: Optional[list[int]],      # [1, 9, 17, 25, 33]
) -> torch.Tensor:
    offset = 1                              # HF hidden_states[0] 是 embedding 输出
    selected_states = []
    for layer_id in layer_ids:
        selected_states.append(hidden_states[layer_id + offset])
    target_hidden = torch.cat(selected_states, dim=-1)  # [B, S, 5×2560] = [B, S, 12800]
    return target_hidden

关键细节：offset=1 因为 HuggingFace 的 output.hidden_states[0] 是 embedding 层输出，[1] 才是第 0 层 Transformer 输出。

A.4.2 Qwen3DFlashDecoderLayer

class Qwen3DFlashDecoderLayer:
    self.self_attn = Qwen3DFlashAttention(config, layer_idx)
    self.mlp = Qwen3MLP(config)                    # 直接复用 Qwen3 的 MLP

    def forward(self, target_hidden, hidden_states, ...):
        residual = hidden_states
        hidden_states = self.input_layernorm(hidden_states)
        hidden_states = self.self_attn(hidden_states, target_hidden=target_hidden)
        hidden_states = residual + hidden_states           # 残差
        residual = hidden_states
        hidden_states = self.post_attention_layernorm(hidden_states)
        hidden_states = self.mlp(hidden_states) + residual
        return hidden_states

A.5 KV 注入注意力机制详解

传统自回归注意力： DFlash KV 注入注意力： Q = proj(hidden) Q = proj(draft_hidden) ← 只来自草稿 K = proj(hidden) K = proj(target_hidden) ⊕ proj(draft_hidden) ← 拼接！ V = proj(hidden) V = proj(target_hidden) ⊕ proj(draft_hidden) ← 拼接！ 因果掩码 (causal) 非因果 (is_causal=False) ← 双向！

forward() 关键步骤

# hidden_states: [B, 16, 2560] — draft    target_hidden: [B, ctx_len, 2560]
q = self.q_norm(self.q_proj(hidden_states))  # [B, 32, 16, 128]

k_ctx = self.k_proj(target_hidden)           # [B, ctx_len, 8*128]
k_noise = self.k_proj(hidden_states)         # [B, 16, 8*128]
k = torch.cat([k_ctx, k_noise], dim=1)      # [B, ctx_len+16, 8*128]  ← 序列维拼接
v = torch.cat([v_ctx, v_noise], dim=1)      # 同理
# → attention(Q, K, V, is_causal=False)  每个draft token attend to ALL

设计原理

  1. KV 注入：草稿模型直接看到目标模型中间层特征 — target→draft 信息通道
  2. 非因果：扩散模型并行去噪，块内所有位置需互相可见
  3. 拼接式混合注意力：交叉注意力+自注意力通过 K/V 拼接实现，比独立双注意力更高效

A.6.2 解码循环 (Draft → Verify → Accept)

while start < max_length:
    ## ═══ Phase 1: Draft ═══
    noise_embedding = target.model.embed_tokens(block_output_ids)  # 共享embed
    draft_hidden = self(target_hidden, noise_embedding, is_causal=False)
    draft_logits = target.lm_head(draft_hidden[:, -block_size+1:])  # 共享lm_head
    block_output_ids[:, 1:] = sample(draft_logits)

    ## ═══ Phase 2: Verify ═══
    output = target(block_output_ids, output_hidden_states=True)
    posterior = sample(output.logits, temperature)

    ## ═══ Phase 3: Accept ═══
    acceptance_length = (block_output_ids[:, 1:] == posterior[:, :-1]).cumprod(1).sum(1)
    start += acceptance_length + 1

接受长度计算

draft  = [A, B, C, D, ...]    target = [A, B, C, X, ...]
比较    = [T, T, T, F, ...]   cumprod = [1, 1, 1, 0, ...] → sum = 3
→ 接受 3 个 draft token + 1 个 target 修正 = 一轮产生 4 个 token

无损保证：输出与纯目标模型完全一致（temperature=0）。

15. 3D MRoPE 位置编码机制详解

15.1 MRoPE 原理

Qwen3-4B 使用标准 1D RoPE，position_ids shape 为 (batch, seq_len)。
Qwen3-Omni 使用 3D Multi-dimensional RoPE (MRoPE)，position_ids shape 为 (3, batch, seq_len)， 将 128 维的 head_dim 切分为三段：

维度用途分配的 head_dim频率对数
mrope_section[0] = 24时间维 (Temporal)48 维 (24×2 for sin/cos)编码视频帧顺序、音频时间步
mrope_section[1] = 20高度维 (Height)40 维 (20×2)编码图像/视频的空间高度位置
mrope_section[2] = 20宽度维 (Width)40 维 (20×2)编码图像/视频的空间宽度位置
总计128 维

交错排列 (mrope_interleaved = true)

频率排布从 [TTT...HHH...WWW...] 重排为 [T,H,W,T,H,W,...] 交错模式， 这使得每个 attention head 都能同时感知三个空间维度的位置信息。

15.2 不同模态的 Position ID 分配策略

纯文本 token:
  T维: [0, 1, 2, 3, ...]    H维: [0, 1, 2, 3, ...]    W维: [0, 1, 2, 3, ...]
  → 退化为等效的 1D RoPE

图像 token (如 H=24, W=24 的 patch grid):
  T维: [t, t, t, ..., t]                    // 时间维恒定
  H维: [0, 0, ..., 1, 1, ..., 23, 23, ...]  // 逐行递增
  W维: [0, 1, ..., 23, 0, 1, ..., 23, ...]  // 逐列递增

视频 token (如 T=3 帧, 每帧 H=12, W=12):
  T维: [0,...,0, 25,...,25, 50,...,50]       // 每帧 jump = position_id_per_seconds(13) × seconds_per_chunk(2) = 26
  H维: [0,0,...,11,11,..., 0,0,...,11,11,...]// 空间位置循环
  W维: [0,1,...,11, 0,1,...,11, ...]         // 同上

音频 token:
  T维: [0, 1, 2, ...]                       // 线性递增
  H维: [0, 0, 0, ...]                       // 恒定为 0 (音频无空间维度)
  W维: [0, 0, 0, ...]                       // 恒定为 0

rope_deltas 机制用于跟踪 prefill 与 decode 阶段之间的位置偏移，确保 KV cache 下的位置连续性。

26.4 logits_to_keep 不支持（错误 #3）

错误

RuntimeError: The expanded size of the tensor (1) must match the existing size
(70) at non-singleton dimension 1. Target sizes: [1, 1]. Tensor sizes: [70]

根因：_dflash_generate() 在 prefill 阶段调用 target(..., logits_to_keep=1) 仅返回最后一个 token 的 logits。但 thinker 模型忽略该参数，返回所有位置的 logits。sample(output.logits, temperature) 返回 shape 为 [70] 的张量，赋值到 output_ids[:, num_input_tokens:num_input_tokens+1] 时大小不匹配。

修复：添加 _ThinkerWrapper 类拦截 logits_to_keep 并手动裁切 logits：

class _ThinkerWrapper(torch.nn.Module):
    def __init__(self, thinker):
        super().__init__()
        self._thinker = thinker

    @property
    def model(self):
        return self._thinker.model

    @property
    def lm_head(self):
        return self._thinker.lm_head

    def forward(self, *args, **kwargs):
        logits_to_keep = kwargs.pop("logits_to_keep", None)
        output = self._thinker(*args, **kwargs)
        if logits_to_keep is not None and output.logits.shape[1] > logits_to_keep:
            output.logits = output.logits[:, -logits_to_keep:, :]
        return output

Wrapper 同时通过 .model（含 .embed_tokens）和 .lm_head 属性暴露接口，供 _dflash_generate() 用于草稿模型的 embedding 和 logit 投影。

32. C++ OpenVINO DFlash 实现：0% 接受率根因分析

将 DFlash 推测解码移植到 C++ OpenVINO 框架（modeling_qwen3_omni_dflash.cpp）后，初始测试表现为 0% 接受率（所有草稿 logits 为 NaN）。 经过系统诊断，发现存在三个独立的 Bug，全部修复后接受率从 0% 提升至 64–73%。

32.1 诊断方法

  1. 中间张量对比：在 C++ combined draft 模型图中添加 debug 输出节点（debug_noise_embedding, debug_conditioned, debug_context_hidden 等），与 Python 逐层对比。
  2. 精度隔离测试：分别使用 inference_precision=f16 和 f32 编译草稿模型，确认 NaN 仅在 f16 下出现。
  3. 权重验证：直接从 safetensors 加载并对比 thinker.lm_head.weight 与 thinker.model.embed_tokens.weight，发现 99.9% 元素不同。

32.2 Bug #1：lm_head 权重被覆盖（主因）

严重性：致命 — 导致 100% 的草稿预测错误（完全不同的 logits 分布）

现象

  • C++ 草稿模型 logits 值量级约 5–6，而 Python 对应值约 30–40
  • C++ argmax 结果与 Python 完全不同（0/15 匹配），即使中间层输出 (conditioned, context_hidden) 完全一致

根因

Qwen3-Omni 的 safetensors 文件同时存储了 thinker.lm_head.weight（在 model-00001-of-00004.safetensors）和 thinker.model.embed_tokens.weight（在 model-00002-of-00004.safetensors）， 尽管 config.json 中 tie_word_embeddings = true。这两个张量：

  • shape 相同：[151936, 2560]，dtype 相同：bfloat16
  • 但 99.9% 的元素值不同，最大差异 0.209，平均差异 0.017
  • lm_head 的 mean_abs = 0.007，embed_tokens 的 mean_abs = 0.015（相差 2 倍）

在 create_qwen3_5_dflash_combined_draft_model() 中：

// 构建流程：
LMHead head(ctx, "lm_head", &lm_root);
head.tie_to(embed.weight_param());          // ✅ 绑定到 embed_tokens
// ...
load_model(lm_root, target_source, ...);    // ❌ 用 lm_head.weight 覆盖了绑定！

load_model() 遍历 source 中的所有 key，找到 lm_head.weight 后将其绑定到 head 的 weight parameter， 覆盖了之前 tie_to() 建立的到 embed_tokens.weight 的绑定。

修复

if (qwen_cfg.tie_word_embeddings) {
    head.tie_to(embed.weight_param());
    // 当 tie=true 时，跳过 load_model — 避免 safetensors 中的
    // 独立 lm_head.weight 覆盖正确的 tied 绑定
} else if (!target_source.has("lm_head.weight")) {
    // ... fallback to embed_tokens
} else {
    load_model(lm_root, target_source, target_finalizer, options);
}

文件：modeling_qwen3_5_text.cpp，影响 create_qwen3_5_dflash_combined_draft_model() 和 create_qwen3_5_dflash_step_model() 两个函数。

32.3 Bug #2：Draft 模型 f16 推理精度导致 NaN

严重性：致命 — 产生全 NaN logits，无法进行任何有效推测

现象

  • 草稿 logits 全部为 -nan
  • target_hidden 正常（shape [1,22,12800]，sum_abs = 380）
  • Python 使用相同输入不产生 NaN

根因

DFlash 草稿模型的 fc 层执行 target_hidden @ fc_weight^T，其中：

  • target_hidden 维度为 ctx_dim = hidden_size × num_draft_layers = 2560 × 5 = 12800
  • 输入值量级约 42（多层 hidden states 的拼接），权重 mean_abs = 0.145
  • 点积涉及 12800 个乘加运算，结果可达 700–1300

虽然图用 f32 构建（dflash_draft.hpp 注释："Draft is always constructed in FP32 to avoid BF16 instabilities"）， 但编译时使用 inference_precision = f16 导致 OpenVINO 在内部以 f16 执行 MatMul。 f16 最大值约 65504，而 RMSNorm 之前的 conditioned 值已达 600–1300，后续 attention 计算中的 softmax 极易溢出为 NaN。

修复

// 草稿模型单独使用 f32 精度编译
ov::AnyMap draft_compile_cfg = compile_cfg;
draft_compile_cfg[ov::hint::inference_precision.name()] = ov::element::f32;
auto compiled_draft = core.compile_model(combined_draft_model, device, draft_compile_cfg);

环境变量 OV_GENAI_DRAFT_PRECISION=f16 可覆盖此默认值用于实验。

32.4 Bug #3：Draft Position IDs 重叠（次要）

现象

C++ 使用 [0..T-1, T-1..T+B-2]（位置 T-1 重复两次），而 Python 使用连续的 [0..T+B-1]。

修复

// 修改前：[0..T-1, T-1..T+B-2]（重叠）
for (size_t i = 0; i < target_hidden_len; ++i)
    pd[i] = static_cast<int64_t>(i);
for (size_t i = target_hidden_len; i < total_pos; ++i)
    pd[i] = static_cast<int64_t>(target_hidden_len - 1 + (i - target_hidden_len));

// 修改后：[0..T+B-1]（连续）
for (size_t i = 0; i < total_pos; ++i)
    pd[i] = static_cast<int64_t>(i);

33. FC 层 FP16 溢出根因分析与修复

在 §32.3 中记录了 draft 模型使用 inference_precision=f16 编译时 logits 全 NaN 的现象，当时的解决方案是将 draft 模型回退到 f32 编译。 本节深入分析了溢出的根因，并实现了一个无性能损失的永久修复，使 FP16 编译成为可能。

33.1 根因定位

DFlashDraftModel 的 forward() 和 build_context_kv() 中，fc 层执行矩阵乘法：

// dflash_draft.cpp (修复前)
auto conditioned = ops::linear(target_hidden, fc_weight());
//   target_hidden: [batch, seq, 12800]  (ctx_dim = hidden_size × num_hidden_layers = 2560 × 5)
//   fc_weight:     [2560, 12800]
//   输出:          [batch, seq, 2560]  — 归约维度 12800

该矩阵乘法的归约维度为 12800，每个输出元素是 12800 个乘积的累加和。

33.2 实测数据证明

使用 Qwen3-Omni-4B 真实推理数据进行验证：

张量形状abs_maxL2 范数 max
fc.weight[2560, 12800]4.5666.37 (行)

关键发现：
  • FC 输出在 FP32 下 abs_max = 155,831，超过 FP16 最大值 65,504
  • FP16 下产生 inf（4/5.3M 元素溢出，约 0.0001%）
  • inf 值传播至所有下游计算 → logits 全 NaN → 接受率 0%
  • 经过 RMSNorm 后，值域收缩到 abs_max ≈ 9.98，完全在 FP16 安全范围内

33.3 为什么 Qwen3.5 没有这个问题

Qwen3.5 和 Qwen3-Omni 使用 完全相同的 DFlashDraftModel 类（dflash_draft.cpp）， 但 Qwen3-Omni 的 target hidden states 有显著更大的激活值（abs_max ~9152 vs Qwen3.5 的更小值）， 可能是因为多模态架构（音频/视觉特征）贡献了极端激活值。

33.4 修复方案：输入缩放 + RMSNorm 尺度不变性

RMSNorm 具有尺度不变性：对于任意正标量 s，RMSNorm(s·x) = RMSNorm(x)。 这是因为 RMSNorm 的公式为 $\frac{x}{\text{RMS}(x)} \cdot \gamma$，分子分母同时乘以 s 后抵消。

因此，只需在 fc matmul 之前对 target_hidden 乘以缩放因子 1/128：

// dflash_draft.cpp (修复后)
// Scale down target_hidden before fc matmul to prevent FP16 overflow.
// RMSNorm is scale-invariant: RMSNorm(s*x) = RMSNorm(x), so results are exact.
auto conditioned = ops::linear(target_hidden * (1.0f / 128.0f), fc_weight());
auto context_hidden = hidden_norm_.forward(conditioned);

修复效果：

  • FC 输出最大值：155,831 / 128 ≈ 1,217（远小于 FP16 max 65,504）
  • RMSNorm 输出 数学上完全相同（尺度不变性保证）
  • 无性能损失（仅增加一个 Multiply 算子）
为什么 .to(f32) 不起作用？
OpenVINO GPU 插件设置 inference_precision=f16 时，会在编译阶段优化掉显式的 Convert-to-f32 节点。 因此 target_hidden.to(ov::element::f32) 插入的 Convert 算子会被删除，matmul 仍然在 f16 下执行。 缩放方案使用 Multiply 算子，不会被优化掉。

34.4 根因：KV Cache Trim 走 CPU 回退路径

投机解码每步需要 trim target 模型的 KV cache（删除被拒绝的 token）。 trim_kv_cache() 的实现有两种路径：

路径条件操作耗时
GPU zero-copy OV_HAS_VARIABLE_STATE_SET_SHAPE 已定义 state.set_shape(new_shape) — GPU 内存重新解释，无拷贝 <0.3ms
CPU 回退 默认 state.get_state() GPU→CPU → CPU 裁剪 → state.set_state() CPU→GPU ~132ms

CPU 回退路径的开销：

  • Qwen3-Omni thinker 有 36 层 × 2 (K+V) = 72 个 state
  • 每个 state 形状 [1, H, seq_len, head_dim]，随 context 增长
  • 每步将 整个 KV cache 从 GPU 复制到 CPU，裁剪后写回 — 约 200MB+ 的往返传输
  • 每步都执行一次（因为投机解码总有被拒绝的 token 需要 trim）
// utils.cpp — CPU fallback path
for (auto& state : states) {
    ov::Tensor old_tensor = state.get_state();   // GPU → CPU copy (全量!)
    // ... trim ...
    ov::Tensor new_tensor(old_tensor.get_element_type(), shape);
    trimmed_tensor.copy_to(new_tensor);            // CPU memcpy
    state.set_state(new_tensor);                   // CPU → GPU copy (全量!)
}

34.9 剩余瓶颈

优化后每步的耗时分布：

  • target verify: 192ms/步 — 当前最大瓶颈，是 target 模型推理本身的开销，难以进一步优化

理论最优 TPOT（接受率 100%，block_size=16）：

$\text{TPOT}_{理论} = \frac{\text{draft} + \text{verify} + \text{other}}{B} = \frac{42 + 192 + 6}{16} = 15 \text{ms/token}$

当前实际 TPOT（codegen, 56% 接受率, avg_accept=1.31）：

$\text{TPOT}_{实际} = \frac{240}{1.31 + 1} = 104 \text{ms/token}$

即当前效率为理论最优的 $15/104 \approx 14\%$，提升空间主要在接受率（draft 模型质量）。

37.3 根因：INT4 与 Speculative Decoding 的计算特性冲突

INT4 量化的收益来自减少 memory bandwidth（weight size 缩小 4×，从显存加载更快）。 这对标准 AR 推理非常有效，因为单 token forward 是完全 memory-bound 的：

配置TPOTTok/s分析
Baseline FP16 (Case 1)148.58 ms6.73单 token AR，memory-bound
Baseline INT4 (Case 3)73.97 ms13.52INT4 weight 减少 4× bandwidth → 2.01× 加速

但 DFlash speculative decoding 的工作模式完全不同：

  1. Draft model 已经极小（5 层 attention+MLP），FP16 下每步仅 1.46ms，INT4 dequant 的固定开销占比更大
  2. Target verify 是 batch 操作（一次验证 block_size 个 token），计算密度更高，已经接近 compute-bound 区域。 INT4 的 dequant 运算增加了额外计算，但 bandwidth 收益在高 batch 下边际递减
  3. 高频调用放大 overhead：29 draft steps + 37 verify calls = 66 次 inference，每次的 INT4 dequant 开销累积为 37.5ms

37.4 Baseline INT4 vs DFlash INT4：为什么差距如此大

Baseline INT4 通过 weight 量化将 memory bandwidth 需求降到 1/4，单 token AR 推理直接获得 2× 加速。 但 DFlash 的 speculative decoding 通过 batch verify 提高了计算密度（从 memory-bound 向 compute-bound 偏移）， 使得 INT4 的 bandwidth 收益被抵消，而 dequantization 的计算开销成为净负担。

本质上：INT4 优化的是 memory-bound 的单 token 场景，speculative decoding 优化的是通过 batch 提高计算效率—— 两者针对不同瓶颈做优化，同时使用时互相矛盾。

37.6 结论与建议

  1. DFlash benchmark 应以 FP16 配置为主：INT4+INT4 组合在 speculative decoding 下没有性能优势
  2. 如需量化，考虑 target INT4 + draft FP16：draft model 已经很小（~200MB），量化无意义；target 量化可能减少 verify 的 weight 加载时间
  3. INT4 baseline 仍有参考价值：它代表了"不用 speculative decoding、仅用量化加速"的上界（13.52 tok/s）， DFlash FP16 (10.71 tok/s) 尚未超过这个上界，说明当前 AccRate (~57%) 还不够高
  4. 理论盈亏平衡点：要让 DFlash FP16 超过 Baseline INT4，需要 AccRate > 57% × (13.52/10.71) ≈ 72%。 当前仅代码类 prompt (Fibonacci 70.3%) 接近此门槛
